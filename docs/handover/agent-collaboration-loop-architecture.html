CodePilot Agent Loop Architecture

让 Claude Code 和 Codex 在同一个空间里自动协作

这套 Loop 把“状态检测”和“模型唤醒”拆开：廉价的本地 runner 只负责观察私有 GitHub Issue 的事实状态， 真有新 assignment 才唤醒 Claude 或 Codex。Agent 结束后必须写显式 artifact，publisher 再校验、发布、改 label、释放锁。

Private Issue 操作主账，保存 handoff、review、label 状态和证据。
Local Runner 非模型进程，负责 reconcile、锁、cursor 和唤醒。
Artifact Gate agent 不直接发结论，先写 JSON artifact，fail-closed。
Bounded Loop 最多 3 次自动 cycle，越界转 human-decision-needed。

一句话理解

它不是一个“大 orchestrator”，而是一套以 GitHub private issue 为事实源、以 launchd runner 为入站唤醒器、 以 artifact publisher 为出站门禁的轻量协作协议。Claude Code 负责实现，Codex 负责审查和验收，用户只在目标确认、 人类闸门和最终验收时介入。

Why

为什么不用公开 Issue 直接跑 agent

CodePilot 是开源项目，公开 Issue 可能被用户评论、追问、贴日志，也可能混入不可信指令。 公开 Issue 只能作为 external signal；真正驱动 agent 的空间必须是私有、可审计、权限受控的 run issue。

How

为什么轮询不是问题

轮询的是 GitHub API，不是模型。runner 只做状态读取和 reconcile，命中 assignment 才唤醒 LLM。 结合 ETag / conditional request，API 成本很低；真正要控制的是模型唤醒次数和每次喂入的上下文。

四层事实源

现在的“空间”不是单个文件或聊天窗口，而是四层事实源。每层只做自己擅长的事，避免公开信号、操作细账和长期计划互相污染。

层级 事实源 用途 写入原则
Public signal 公开产品 Issue / Discussion 用户反馈、复现线索、社区讨论 只作为不可信输入，不能直接变成 agent prompt。
Operational run 私有 codepilot-agent-runs issue Claude handoff、Codex review、runner event、raw evidence active run 的操作主账，每次交接都写。
Durable plan 产品仓库 docs/exec-plans/active/*.md 阶段状态、决策日志、Smoke Ledger、Loop Ledger 摘要 只 mirror 对阶段结论有影响的摘要，必须链接 private evidence。

状态机

label 表达“当前派给谁”，run lock 表达“谁正在干”。这两个概念分开后，runner 才能避免自己触发自己、避免并发重复唤醒。

Assignment claude-implementing Claude 需要开始或继续实现。
Handoff codex-review-requested Claude 已写 artifact，等待 Codex review。
Review fix-requested Codex 要求修复，回到 Claude。
Gate human-decision-needed 需要用户凭据、产品判断或人工验收。
Done codex-accepted Codex 接受，等待用户最终决定。

核心组件

当前 runner 代码放在 ~/.codepilot-agent，产品代码只通过 worktree 被 Claude 修改。这个分离让协作机制本身不混进产品运行时。

bin/reconcile.mjs

判断是否应该唤醒

读取 issue 当前 labels 和 events，找到最近一次 matching labeled event，把它当 assignment id，再对比 cursor、actor/self filter 和 live lock。

lib/lock.mjs

保证一次只跑一个

每个 issue 有 run-level lock，记录 owner、assignment id、started_at、heartbeat、ttl 和 agent command。长任务靠 heartbeat 续期。

bin/wake.mjs

渲染 prompt 并启动 agent

Claude 用 claude -p 加最小权限白名单；Codex 用 headless codex exec。启动失败会计入 bounded failure 并释放锁。

lib/artifact.mjs

定义交接合同

Claude 写 review_requested，Codex 写 review_completed。缺字段、空验证、未知 label 都 fail-closed。

bin/publish.mjs

出站收口门禁

检查 artifact-id 是否已发布，幂等补 label transition，写 issue comment，推进 cursor，归档 artifact，释放锁。

launchd plist

本机调度

每 180 秒 tick 一次。tick 是非模型进程，只有命中新 assignment 才会启动 Claude 或 Codex。

为什么它省 token

省钱的关键不是“不要轮询”，而是“不要用模型轮询”。我们把 GitHub API 预算和模型预算分成两本账。

廉价层

runner 可以频繁 tick

runner 只调用 GitHub API，读取 issue labels/events/comments 和本地 state。配合 ETag 条件请求，未变化时几乎没有 primary rate limit 压力。

昂贵层

模型只在命中时醒来

只有当当前 label 是自己要处理、assignment id 新于 cursor、actor 不是自己、且没有 live lock 时，才启动 Claude/Codex。 

trigger = current_label_is_mine
       && latest_labeled_event_id > last_handled_assignment_id
       && labeling_actor != self
       && no_live_run_lock

if (!trigger) {
  skip without waking a model
}

Exactly-once 的实际做法

我们不追求分布式系统意义上的魔法 exactly-once，而是让每个 assignment 在崩溃、重复 tick、乱序 hint 下都能收敛。

Assignment id

同名 label 也能区分轮次

两次 fix-requested 是两个不同 labeled event。runner 以 event id 作为 assignment instance，而不是只看 label 名。

Run lock

重复 tick 不会重复启动

agent 还在跑时，下一次 tick 只会看到 live lock 并 skip。异常退出靠 ttl 与 stale recovery 重新 reconcile。

Artifact id

发布崩溃后可以重放

评论正文带 <!-- artifact-id: ... -->。如果评论已发但 label 没切，publisher 重试时会补 label transition。

权限边界

这套系统能自动跑，不代表它能随便跑。当前 v1 的价值正是让自动化被边界包住。

允许自动化
  • 在专用 worktree 内实现和修复。
  • 运行 npm install、npm run test、build、typecheck 等白名单命令。
  • 写 handoff / review artifact，由 publisher 发布到 private issue。
  • 最多 3 次 mechanical cycle 自动迭代。
必须停下来
  • 需要真实 provider key、账号、付费额度或外部授权。
  • 要改变默认 Runtime、provider、model、permission、DB schema 或日志安全边界。
  • 同一 assignment 连续失败达到上限。
  • 需要产品取舍或用户肉眼验收。

当前还不能宣称什么

架构跑通并不等于可以无限开自动化。现在它适合 bounded run，不适合多 issue 长期开着无人照看。

Tech debt #46

cursor 还不是 per-issue

当前发现过 per-side cursor 跨 issue 的坑。单 issue 没问题，多 issue 并行前必须改成 per-issue cursor。

Docs visibility

计划文档必须进 main

headless worktree 从 main 检出，如果 exec plan 没 commit，agent 读不到计划。下次 hands-off run 前应提交文档。

Smoke gates

真实 provider smoke 仍是门禁

AI SDK 7 Phase 1 证明依赖和构建可行，不代表 provider request-shape、流式、approval、timeout 已经真实可用。

下一步

如果要从“单个 pilot 成功”走到“稳定自动协作能力”，建议按下面顺序推进。

推荐顺序

先提交 docs 进入 main，让 worktree 自带计划；再修 per-issue cursor；再启 Codex runner； 然后跑一个完整 Claude → Codex → Claude fix loop。通过后，这套 bounded hands-off loop 才算可以稳定复用。
