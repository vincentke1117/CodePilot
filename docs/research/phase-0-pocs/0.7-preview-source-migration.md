## [设计]

### PreviewSource discriminated union（完整 TS 类型）

建议直接写进 `src/hooks/usePanel.ts`（或独立 `src/types/preview.ts` 再 re-export，取决于是否想让非 panel 模块也复用；本 POC 推荐就放在 `usePanel.ts` 以减少文件新增噪音）：

```ts
/**
 * 预览面板的内容来源 — discriminated union。
 *
 * - `file`:            基于工作目录内文件路径，由 /api/files/preview 拉取
 * - `inline-html`:     直接传入 HTML 字符串，走 iframe srcDoc 渲染（Phase 2.1）
 * - `inline-jsx`:      直接传入 JSX/TSX 源码，走 Sandpack 渲染（Phase 2.1 / 2.2）
 * - `inline-datatable`: 结构化表格数据，走 DataTable viewer（Phase 5.4）
 *
 * 所有 inline-* 变体可选带 `virtualName`，用于 header 标题 / breadcrumb / copy 的兜底文本。
 */
export type PreviewSource =
  | { kind: 'file'; filePath: string }
  | { kind: 'inline-html'; html: string; virtualName?: string }
  | {
      kind: 'inline-jsx';
      jsx: string;
      virtualName?: string;
      /** 可选 runtime 绑定，Phase 2.1 Sandpack 决定是否注入 */
      bindings?: Record<string, unknown>;
    }
  | {
      kind: 'inline-datatable';
      rows: unknown[][];
      header: string[];
      virtualName?: string;
    };
```

### `PanelContextValue` 改动

### `AppShell` adapter 代码草案

替换 `src/components/layout/AppShell.tsx:355-366` 的实现。核心思路：**state 唯一来源变为 `previewSource`**，`previewFile` / `setPreviewFile` 是派生视图 + 包装函数，保持外部 API 兼容。

```ts
// 单一 state（替代原 previewFile useState）
const [previewSource, setPreviewSourceRaw] = useState<PreviewSource | null>(null);
const [previewViewMode, setPreviewViewMode] = useState<PreviewViewMode>("source");

// 派生视图 — 现有 PanelZone / FileTreePanel 等继续读 previewFile 零改动
const previewFile: string | null =
  previewSource?.kind === 'file' ? previewSource.filePath : null;

// 主 setter：同时联动 previewOpen + previewViewMode（inline 默认 rendered）
const setPreviewSource = useCallback((source: PreviewSource | null) => {
  setPreviewSourceRaw(source);
  if (source) {
    // file 沿用原 defaultViewMode；inline-html/inline-jsx 强制 rendered
    //（inline-datatable 的 viewMode 不重要，但保持 rendered 以便未来扩展）
    if (source.kind === 'file') {
      setPreviewViewMode(defaultViewMode(source.filePath));
    } else {
      setPreviewViewMode('rendered');
    }
    setPreviewOpen(true);
  } else {
    setPreviewOpen(false);
  }
}, []);

// 兼容 adapter：老调用 setPreviewFile(path) 内部转成 setPreviewSource
const setPreviewFile = useCallback((path: string | null) => {
  if (path === null) {
    setPreviewSource(null);
  } else {
    setPreviewSource({ kind: 'file', filePath: path });
  }
}, [setPreviewSource]);

// 路由切换 effect 保持原行为（src/components/layout/AppShell.tsx:370-373）
useEffect(() => {
  setPreviewSourceRaw(null);
  setPreviewOpen(false);
}, [pathname]);
```

并在 `useMemo` 返回值里同时导出 `previewSource` / `setPreviewSource` / `previewFile` / `setPreviewFile`，依赖数组加 `previewSource`、`setPreviewSource`。

### 约束检查（让 TypeScript 帮忙）

- 所有 `PreviewSource` 分支都带 `kind` 字面量，调用 `switch(source.kind)` 时 exhaustiveness 可被 TS 强制（建议在 `PreviewPanel` 的渲染 switch 里配 `const _exhaustive: never = source;` 兜底分支）
- `setPreviewFile` 参数仍是 `string | null`，外部零改动
- `previewFile` 读取端拿到的仍是 `string | null`；不能把 inline-* 漏判成有路径

## [风险与注意事项]

### R1. PanelZone gate 漏改（最高风险）

执行计划 `:60` / `:376-378` 已明确标注。本 POC 再次强调：**两处（L15 + L22）必须同时改**，且 `anyOpen` 和条件挂载要一起改，避免某个分支漏 inline-* 导致行为不一致。建议 Phase 1.5 PR 里对这两行加 code comment 显式说明「必须用 previewSource 而非 previewFile」。

### R2. `PreviewPanel` 渲染分支大量基于扩展名判断，inline-* 没有扩展名

`canRender` / `isMedia` / `isHtml` 三个辅助（`src/components/layout/panels/PreviewPanel.tsx:59-82`）都是 `getExtension(filePath)` 查表，对 inline-* 无意义。Phase 1.5 要在顶层按 `previewSource.kind` 分派后再走各自专用渲染路径；**不要**试图给 inline 构造假的 `filePath = 'preview.html'` 复用老逻辑 — 会被 MediaView 之类误判。

### R3. `useEffect(loadPreview)` guard

`src/components/layout/panels/PreviewPanel.tsx:106` 已经跳过 `isMediaPreview(filePath)` 和空字符串；Phase 1.5 要**扩展 guard**：只有 `previewSource?.kind === 'file'` 才发请求。inline-* 场景如果漏 guard，会拿 `filePath = ''` 去打 API，可能收到 400（API 要求 path 非空），造成无害但嘈杂的错误日志。

### R4. `ViewModeToggle` 对 inline-* 的语义

`src/components/layout/panels/PreviewPanel.tsx:187-189` 的 `ViewModeToggle` 只在 `canRender && !isMedia` 时显示。`canRender` 对 inline-* 永远 false（因为基于扩展名）。

- `inline-html`: 建议强制 `rendered`，隐藏 toggle（iframe 预览为主；源码预览对普通用户价值低）
- `inline-jsx`: 同上，Sandpack 主导
- `inline-datatable`: 无 source/rendered 概念，toggle 不显示

Phase 1.5 实施时需重写 toggle 的显示条件，不能直接复用 `canRender && !isMedia`。

### R5. `FileTreePanel` toggle 语义与 adapter 的互动

`FileTreePanel.tsx:48-55` 比较 `previewFile === path` 决定是否「再次点击同文件关闭」。当面板正处于 inline-* 状态时，派生 `previewFile` 为 `null`，`previewFile === path` 必然 false，所以走 else 分支→切到 file kind。这是**正确行为**：用户在预览 Artifact 的 inline 状态下，点文件树某个文件，意图是看那个文件，不是再次 toggle 关闭。**不需要改 FileTreePanel**。

### R6. 路由切换的清理副作用

`AppShell.tsx:370-373` 的 `useEffect(..., [pathname])` 当前清 `previewFile`。迁移后要改为清 `previewSource`（一个 setter 就够）。**注意**：不要忘了同时清 `previewOpen`（原代码显式清了 `setPreviewOpen(false)`），这是防御性写法，防止两个状态失配。

### R7. `useMemo` 依赖数组

`AppShell.tsx:452` 加两个新字段 `previewSource, setPreviewSource` 到依赖数组。旧字段 `previewFile, setPreviewFile` 建议保留（它们现在是派生值，但保留可以让 React 内部比对更安全；ESLint 也会提示）。否则可能遇到 stale closure。

### R8. 序列化 / 持久化

目前 `previewFile` 不持久化（路由切换就清了）。`previewSource` 同策略：**不持久化**，刷新后恢复到 null 即可。inline-* 内容（尤其 `inline-html`/`inline-jsx`）可能很大，存 localStorage 有安全和尺寸风险。Phase 1.5 验收标准里不包含持久化。

### R9. `setPreviewSource(null)` 与旧 `setPreviewFile(null)` 等价性

老代码常连调 `setPreviewFile(null) + setPreviewOpen(false)`（例：`PreviewPanel.tsx:152-153`、`FileTreePanel.tsx:49-50`）。迁移后的 `setPreviewSource(null)` 已在 AppShell adapter 内部联动 `setPreviewOpen(false)`，因此调用点可简化为单调用；但**显式双调用不算错**，只是冗余。Phase 1.5 一并清理。

### R10. adapter 的单向性

`setPreviewFile(path)` → 通过 adapter 内部永远 produce `{ kind: 'file' }`。**反向不通**：无法通过 `setPreviewFile` 设置 inline-*。这是期望行为 — 旧 API 仅支持 file kind，新 API 必须直接用 `setPreviewSource`。文档和 JSDoc 要注明这点，避免后续接手人以为 `setPreviewFile` 已弃用可以删。

### R11. grep 覆盖度验证

本 POC grep 全仓确认的 `previewFile`/`setPreviewFile` 读写点仅 5 个文件（`usePanel.ts`、`AppShell.tsx`、`PanelZone.tsx`、`PreviewPanel.tsx`、`FileTreePanel.tsx`）；`previewOpen`/`setPreviewOpen` 同上加 AppShell 自身的 useState。`electron/` 和 `apps/` 目录无任何匹配。测试目录（`tests/`）不存在。Phase 1.5 实施前建议重跑一次 grep 作为 sanity check（计划变更后可能有新消费点）。

### R12. 验收 smoke test 建议
